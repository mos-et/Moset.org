# ESPECIFICACIÓN TÉCNICA MAESTRA: LENGUAJE MOSET
**Documento Fundacional - Parte 1 de 2: Capa Sintáctica y Gramatical**

## 1. IDENTIDAD Y TOPOLOGÍA BASE
* **Extensión de Archivo:** `.et` (Reconocimiento MIME exclusivo para el compilador Moset).
* **Delimitación de Bloques (Scope):** El lenguaje elimina por completo el uso de llaves `{}`. Las fronteras semánticas se definen mediante una estricta topología de espacios en blanco (indentación).
* **Nomenclatura de Variables:** El Lexer exige formatos contiguos (ej. `camelCase` o `snake_case`) utilizando el espacio en blanco como separador absoluto de tokens.

## 2. TABLA LÉXICA Y SISTEMA DE MACROS (LSP)
Moset utiliza una semántica visual basada en "caritas" formadas por caracteres especiales. Para garantizar la usabilidad, estas se generan mediante macros de expansión confirmadas por la tecla `TAB`.

| Intención Lógica | Macro (Tipeo) | Símbolo Final | Comportamiento del Motor (AST) |
| :--- | :--- | :--- | :--- |
| **Función / Rutina** | `,,` + `TAB` | `:,]` | Define el inicio de un bloque ejecutable aislado. Los corchetes evitan colisiones con paréntesis matemáticos. |
| **Excepción / Catch** | `,,,` + `TAB` | `:,[` | Define el plan de contingencia (Plan B). Aisla el error si la función base falla. |
| **Asincronismo / Await** | `...` + `TAB` | `:,\` | Pausa la ejecución del hilo actual esperando la resolución de una promesa (APIs, DBs) sin bloquear el CPU. |
| **Comentario** | N/A | `:@` | Operador de silenciamiento. El compilador ignora la línea entera durante el escaneo (salto de instrucción). |

*Nota sobre el manejo de Errores (`:,[`):* Admite una **arquitectura dual**. Puede usarse como bloque estructurado debajo de una función, o "En Línea" para asignaciones inmediatas.
Ejemplo En Línea: `resultado = calcularTotal() :,[ "Error de cálculo"`

## 3. EXPRESIONES Y FLUJO DE CONTROL
Moset abandona el paradigma declarativo pesado a favor de un flujo de evaluación directa.

* **Retorno Implícito:** Se elimina la necesidad de la palabra reservada `return` o `devolver`. La última línea evaluada lógicamente dentro de una función `:,]` es el valor que el motor escupe y entrega por defecto.
* **Condicionales como Expresiones:** Los bloques lógicos no solo bifurcan el camino, sino que retornan valores, permitiendo asignaciones directas en una sola línea sin declarar variables nulas previamente.
  * *Ejemplo:* `estado = si saldo > 0: "Aprobado" sino: "Rechazado"`

## 4. ESTRUCTURAS DE DATOS Y COLECCIONES
Diseñadas para maximizar la velocidad de acceso y la lectura humana, alejándose de las clases pesadas.

* **Agrupación Lógica (Moldes):** Reemplazo directo de la Programación Orientada a Objetos. Los `moldes` son estructuras de datos (Structs) atómicas que empaquetan variables sin la burocracia de la herencia profunda.
  * *Sintaxis:* `molde Usuario: nombre, rol`
* **Indexación Humana (Base 1):** Las listas, arreglos y colecciones comienzan a contar desde el índice `1` (no desde `0`). El compilador realiza el offset matemático hacia el hardware en segundo plano (Zero-Cost Abstraction). Se permite el acceso tradicional por corchete `lista[1]` o semántico `lista.primero`.
* **Motor de Iteración Dual:** Las colecciones soportan dos paradigmas de recorrido para adaptarse a la complejidad de la tarea:
  1. *Paradigma Funcional (En línea):* Para transformaciones rápidas. Ej: `precios.aplicar(transformacion)`
  2. *Paradigma Estructurado (Bloque):* Para flujos complejos. Ej: `por cada precio en precios:`

1. NÚCLEO DE PROCESAMIENTO Y ABSTRACCIÓN IDIOMÁTICA
Árbol Sintáctico Universal (U-AST): El núcleo de la revolución de Moset. Los archivos .et no almacenan código en un idioma humano específico (inglés o español), sino que guardan la "intención matemática" pura mediante tokens abstractos inmutables (ejemplo: [TOKEN_CONDICION]).
Motor de Renderizado Dinámico: El entorno de desarrollo, a través del Language Server Protocol (LSP), lee el U-AST y traduce visualmente los tokens al idioma configurado por el programador en tiempo real.
Resolución de Enlaces Bilingüe (Imports): El sistema de dependencias utiliza metadatos. Si se importa una librería extranjera, las variables y funciones públicas se mapean al diccionario local del desarrollador sin alterar el repositorio original, destruyendo la barrera del inglés técnico obligatorio.
2. COMPILACIÓN Y MÁQUINA VIRTUAL
Arquitectura Híbrida de Ejecución: Para asegurar un rendimiento apto para orquestadores masivos, Moset no es un lenguaje puramente interpretado basado en texto. El motor procesa el AST universal y lo compila al vuelo en un bytecode optimizado.
Portabilidad Absoluta: La Máquina Virtual de Moset asegura que ese bytecode corra de manera idéntica y a máxima velocidad tanto en servidores pesados en la nube como en hardware de recursos limitados o procesadores ARM.
3. HERRAMIENTAS DE CONSOLA (CLI) Y DESPLIEGUE BARE-METAL
El Comando Global moset: El punto de entrada central para todo el sistema operativo.
Ejecución Directa: moset run archivo.et levanta el intérprete y ejecuta el script inmediatamente.
Modo Interactivo (REPL): Ejecutar moset a secas abre una consola viva y persistente. Permite evaluar expresiones, probar el comportamiento de funciones y testear asincronismo línea por línea antes de guardar el código en un archivo.
Gestor de Paquetes Nativo: Mediante el comando moset instalar <nombre_paquete>, el CLI se conecta al repositorio oficial para descargar dependencias, librerías y módulos de terceros, unificando la distribución de código de la comunidad.
Despliegue Directo por SSH: El motor central del lenguaje permite la clonación de su código fuente mediante Git y llaves criptográficas (ej. git clone git@.../moset-core.git). Esto garantiza transparencia total, seguridad en la infraestructura y facilita la compilación manual.
4. ECOSISTEMA DE DESARROLLO E INTELIGENCIA ARTIFICIAL
IDE Oficial (Moset.org): Un fork nativo, de instalación rápida y "Fricción Cero". El motor de traducción U-AST viene embebido de fábrica, permitiendo que cualquier persona comience a programar en su idioma materno al instante.
Extensión de Entorno Universal: Un plugin silencioso para editores veteranos que intercepta los gatillos de macros (el sistema de Tabulador para las caritas :,], :,[, :,\) y conecta el analizador de sintaxis al vuelo.
Asistencia IA Integrada (Efecto Cline): Al contar con una sintaxis ultra-estricta (basada en el U-AST y delimitación por espacios contiguos), Moset resulta ser el lenguaje más predecible del mercado para un modelo fundacional de IA. El ecosistema incluye un súper agente nativo capaz de predecir arquitecturas, auto-completar lógicas de moldes y gestionar el asincronismo sin alucinaciones semánticas.

¡Te explicás a la perfección! Acabás de describir uno de los conceptos más profundos en la teoría de compiladores: la **Tokenización** combinada con una **Tabla de Símbolos Bilingüe**. 

Tu lógica va exactamente en la dirección correcta. Estás separando la "representación visual" (lo que el humano lee y escribe) del "significado semántico" (lo que la máquina entiende). 

Sin embargo, para que el motor de Moset en Rust sea ridículamente rápido y consuma la menor memoria posible, podemos optimizar tu idea quitando el "salto" intermedio (que el 18 llame al 15). 

Aquí te muestro cómo se implementaría esto a nivel de "puro fierro" en el núcleo de tu compilador:

### 1. El Enfoque Optimizado para Palabras Reservadas
En lugar de que el español apunte a un número que luego apunta a otro número, hacemos que **todos los idiomas apunten directamente al Token Maestro** en el U-AST. 

En Rust, esto no se hace con números puros, sino con una estructura ultra-eficiente llamada `Enum` (Enumeración). Tu "15" sería algo como `Token::Condicional`.

* **Si el IDE está en Inglés:** El Lexer lee `if` ➔ Lo convierte instantáneamente en `Token::Condicional`.
* **Si el IDE está en Español:** El Lexer lee `si` ➔ Lo convierte instantáneamente en `Token::Condicional`.

El Árbol Sintáctico (U-AST) solo recibe `Token::Condicional`. No le importa en qué idioma se escribió; ejecuta la lógica a máxima velocidad. Cuando el LSP tiene que mostrarle el código a otro programador, hace la inversa: lee `Token::Condicional` y lo dibuja en pantalla como `if` o `si` según la configuración del usuario.

### 2. El Enfoque para Variables de Usuario (Donde tu idea brilla)
Para las palabras reservadas del lenguaje (`si`, `mientras`, `molde`), el mapeo es estático. Pero el verdadero desafío arquitectónico que plantea Moset es: ¿Qué pasa con las variables y funciones inventadas por el usuario?

Aquí es donde tu idea de los números cobra todo el sentido del mundo mediante **Identificadores Únicos (IDs)**.

Supongamos que un desarrollador crea una función de física.
1. El programador escribe en español: `molde Gravedad: fuerza`
2. El compilador de Moset **no guarda la palabra "Gravedad" en el código fuente**.
3. En su lugar, el motor genera un ID numérico (o un hash) invisible. El archivo `.et` estructuralmente guarda algo como: `[MOLDE_ID_8492]: [VAR_ID_102]`
4. En paralelo, Moset crea un archivo de metadatos ligero (el "diccionario") que dice:
   * `[MOLDE_ID_8492]` = "Gravedad"
   * `[VAR_ID_102]` = "fuerza"

Cuando un programador estadounidense importa esa librería, su entorno lee los IDs matemáticos puros (`8492` y `102`) y busca en el diccionario local o en la nube para renderizarlos como `Gravity` y `force`. El código nunca se rompe porque la lógica depende de los IDs, no de las letras.

8. MOTOR DE EJECUCIÓN VISUAL (MosetOutputPanel — Fase 12)
───────────────────────────────────────────────────────────
El comando Tauri "ejecutar" retorna JSON estructurado con tipos de línea:

  { "lines": [
    { "type": "header",   "content": "═══ Resultado ═══" },
    { "type": "quantum",  "content": "Bit:[0.75]", "value": 0.75 },
    { "type": "molde",    "content": "Usuario { nombre: Ana, edad: 30 }" },
    { "type": "error",    "content": "Variable no definida: x" },
    { "type": "text",     "content": "Hola Mundo" },
    { "type": "separator","content": "───────────────" }
  ]}

  classify_output_line() en lib.rs detecta el tipo de cada línea de output
  del Motor para que el frontend lo renderice con su componente visual.

  Arquitectura de eventos:
    window.dispatchEvent(new CustomEvent("run-moset-code", {
      detail: { codigo: "...", filename: "archivo.et", language: "moset" }
    }))
  El handler en App.tsx invoca directamente al Motor via Tauri invoke().
